# Next.js
## 1. El pecado original de React: Las SPA (Single Page Applications)

React nació para crear interfaces de usuario dinámicas e interactivas. El flujo tradicional de React "puro" (usando Vite o el viejo Create React App) funciona bajo el modelo de **Client-Side Rendering (CSR)**.

¿Cómo funciona esto?

1. El usuario entra a tu web.
2. El servidor le devuelve un archivo HTML prácticamente **vacío** (un `<div>` con un `id="root"`) y un archivo JavaScript gigante.
3. El navegador del usuario descarga ese JS, lo ejecuta, "construye" la página y recién ahí el usuario ve algo.

### ¿Cuáles son los problemas de este enfoque en el mundo real?

* **SEO desastroso:** Cuando los robots de Google (crawlers) entran a tu sitio para indexarlo, solo ven un HTML vacío. Si tu negocio depende de aparecer en Google (un e-commerce, un diario, una landing page), estás muerto.
* **Performance (FCP inicial bajo):** Si el usuario tiene una conexión móvil lenta, va a ver una pantalla en blanco durante varios segundos mientras se descarga y procesa todo el JavaScript.
* **Cascadas de Fetching (Data Waterfall):** Tenés que renderizar el componente, meter un `useEffect`, disparar un `fetch` a la API, esperar que vuelva, setear el estado y volver a renderizar. Todo esto ocurre en el cliente, multiplicando el tiempo de carga.

---

## 2. ¿Qué es Next.js y qué resuelve?

Next.js es un **framework de React para producción** creado por la empresa Vercel. Su propósito principal es **mover el trabajo pesado del cliente (el navegador) al servidor**.

En lugar de obligar al navegador del usuario a compilar toda la página desde cero usando JavaScript, Next.js se encarga de renderizar los componentes de React en el servidor y entregarle al navegador un **HTML ya pre-construido, con el contenido listo para ser visto.**

### Los 3 pilares que resuelve Next.js:

* **Estrategias de Renderizado Flexibles (La joya de la corona):** Te permite elegir cómo querés que se renderice cada ruta de tu aplicación. ¿Querés que se genere una sola vez cuando compilas la app (**SSG - Static Site Generation**)? ¿Querés que se genere en el servidor en tiempo real para cada usuario (**SSR - Server-Side Rendering**)? ¿O querés un híbrido moderno usando **Server Components**? Podés combinar todo en el mismo proyecto.
* **Ruteo Basado en Archivos (File-system Routing):** Se terminó el sufrimiento de configurar `react-router-dom` con cientos de líneas de código espejo. En Next.js (especialmente con el **App Router** actual), la estructura de tus carpetas define las rutas de tu aplicación (ej. `app/productos/page.js` automáticamente se convierte en la URL `/productos`).
* **Optimización Nativa Out-of-the-Box:** Next.js incluye componentes ultra-optimizados para Imágenes (`<Image>`), Fuentes (`next/font`) y Scripts. El framework automáticamente hace compresión, conversión a formatos modernos (como WebP), pre-carga de páginas (prefetching) y code-splitting (romper el JS en pedazos chicos para cargar solo lo que el usuario necesita).

---

## 3. ¿Para qué fue pensado y para qué se usa actualmente (2026)?

Originalmente, Next.js nació para resolver el problema del SEO y los sitios estáticos o de noticias. Pero hoy, el panorama cambió radicalmente. Con la maduración del **App Router** y los **React Server Components (RSC)**, Next.js se convirtió en un framework **Full-stack**.

Hoy en día, se utiliza para:

* **E-commerce a gran escala:** Donde necesitás velocidad absurda para no perder ventas, SEO perfecto para indexar miles de productos y actualizaciones en tiempo real de stock (ej. combinando renderizado estático con revalidación bajo demanda).
* **Aplicaciones Web Complejas (SaaS):** Plataformas de gestión, dashboards o herramientas donde la seguridad es crítica. Al ejecutar código en el servidor, podés conectarte directamente a la base de datos o consultar APIs privadas usando tokens ocultos, sin exponer jamás tus credenciales al navegador del cliente.
* **Sitios de Contenido y Portales de Noticias:** Donde el contenido cambia constantemente y se necesita una velocidad de respuesta inmediata para millones de usuarios simultáneos.

> 💡 **En resumen:** Next.js no reemplaza a React; potencia a React. Transforma una librería de interfaz de usuario en una arquitectura web completa, dándote las herramientas para controlar exactamente dónde y cuándo se procesa tu código (Servidor vs. Cliente) para lograr la máxima eficiencia posible.

## App Router
**App Router** es dejar atrás el Next.js "viejo" (el Pages Router que todavía enseñan en cursos desactualizados).

Para entender la arquitectura del App Router, tenemos que hablar de su cimiento fundamental, que es un cambio de paradigma total en el ecosistema de React: los **React Server Components (RSC)**.

Olvidate por un segundo de las carpetas. Vamos a entender primero la arquitectura conceptual.

## 1. El Cambio de Paradigma: Client vs. Server Components

Antes, en React, **todo** el árbol de componentes se ejecutaba en el navegador del cliente. Con el App Router, tu aplicación se divide en dos mundos que coexisten perfectamente:

```
                  [ SERVIDOR ]  (Node.js / Edge Runtime)
                       │
         ┌─────────────┴─────────────┐
         ▼                           ▼
  Server Component            Server Component
         │                           │
         └─────────────┬─────────────┘
                       │  (Pasa datos/HTML vía streaming)
                       ▼
                  [ CLIENTE ]   (Navegador del usuario)
                       │
                       ▼
                Client Component (useState, useEffect, etc.)

```

### Server Components (Por defecto)

En el App Router, **todos los componentes dentro de la carpeta `app/` son Server Components por defecto**.

* **¿Qué significa?** Que el código de ese componente se ejecuta *únicamente* en el servidor. El JavaScript de ese componente **nunca** viaja al navegador del usuario.
* **¿Qué ganamos?** Podés hacer un `fetch` directo a tu base de datos, usar contraseñas o tokens privados, y el tamaño del bundle de JavaScript que le llega al cliente es literalmente **cero bytes** para ese componente.

### Client Components

Son los componentes de React tradicionales a los que estás acostumbrado. Para usarlos, tenés que poner la directiva `"use client"` en la primera línea del archivo.

* **¿Cuándo se usan?** Únicamente cuando necesitás interactividad: usar hooks de estado (`useState`, `useReducer`), hooks de ciclo de vida (`useEffect`) o APIs del navegador (como `window.addEventListener` o `localStorage`).

> ⚠️ **Error de Novato:** Muchos creen que `"use client"` significa que el componente *solo* se ejecuta en el cliente. **Falso.** Los Client Components también se pre-renderizan en el servidor para generar el HTML inicial, y luego se "hidratan" (se les inyecta la interactividad) en el navegador.

## 2. La Anatomía del Sistema de Archivos (File-system Routing)

El App Router utiliza las carpetas para definir las URLs, pero lo brillante es cómo utiliza **archivos especiales con nombres reservados** para aplicar patrones de diseño estructurales (árboles de componentes) de forma automática.

Si creás una ruta como `app/dashboard/`, podés meter adentro los siguientes archivos clave:

### A. `layout.js` (Estructura y Persistencia)

Define la interfaz que se comparte entre varias páginas (por ejemplo, una barra lateral o un menú de navegación).

* **Comportamiento Senior:** Cuando el usuario navega entre subrutas (ej: de `/dashboard/perfil` a `/dashboard/ajustes`), el `layout.js` **no se vuelve a renderizar**. Mantiene su estado. Esto evita re-renders innecesarios y ahorra cómputo.

### B. `page.js` (El Contenido Único)

Es la UI única de esa ruta específica. Es lo que se renderiza dentro del `layout.js` como su propiedad `children`.

### C. `loading.js` (Streaming Instantáneo)

Este es uno de los puntos más fuertes de la arquitectura actual. Next.js utiliza por detrás `Suspense` de React. Si creás un archivo `loading.js`, Next.js va a enviar **inmediatamente** el HTML del `layout.js` junto con tu pantalla de carga (un esqueleto, un spinner) al navegador. Mientras tanto, en el servidor, se siguen resolviendo las promesas de datos del `page.js`. Cuando los datos están listos, el servidor "inyecta" el contenido real. El usuario siente que la página carga instantáneamente.

### D. `error.js` (Aislamiento de Fallos)

Define un "Error Boundary" de React de forma automática para esa ruta. Si algo rompe en `page.js` (un error de API, un objeto `undefined`), **no se cae toda la aplicación**. El error se contiene dentro de esa sección y se muestra la UI de `error.js`, permitiendo que el resto de la página (como el menú de navegación) siga funcionando perfectamente.

## 3. ¿Por qué el mercado laboral exige saber esto? (Buenas Prácticas)

En el mercado actual, las empresas buscan optimizar costos de servidores y mejorar métricas de Core Web Vitals (como el LCP - Largest Contentful Paint). La arquitectura del App Router te obliga a seguir la regla de oro: **"Lleva la lógica lo más cerca posible de los datos".**

* **Mala práctica de curso básico:** Poner `"use client"` arriba de todo el layout o de la página principal porque "así me funcionan los hooks y no me complico". Esto destruye los beneficios de Next.js y volvés a tener una SPA pesada.
* **Buena práctica Senior:** Mantener las páginas y layouts como Server Components. Hacer el fetching de datos ahí arriba (en paralelo o en secuencia, tema que veremos más adelante). Y si un botón específico necesita un `useState` para abrir un modal, **aislás** ese botón en un componente hijo chiquito llamado `<ModalButton />` y a *ese* solo le ponés `"use client"`.

De esta forma, tu página sigue siendo 95% estática y ultrarrápida, y solo un 5% requiere JavaScript interactivo en el cliente.

## Pasaje de datos entre servidores y clientes

El pasaje de datos entre servidores y clientes es el "talón de Aquiles" de muchos desarrolladores que saltan de React a Next.js sin entender los fundamentos. Desarmemos cómo funciona este puente, qué restricciones existen por diseño arquitectónico y cuáles son las mejores prácticas en el mercado actual.

---

## 1. La regla de oro: El flujo es unidireccional (De arriba hacia abajo)

En la arquitectura de Next.js, los datos viajan en un solo sentido: **del Servidor al Cliente**.

Un **Server Component** puede importar, renderizar y pasarle datos (props) a un **Client Component**. Pero un Client Component **no puede** importar directamente un Server Component para meterlo dentro de su JSX de la forma tradicional, porque el cliente no puede ejecutar código de servidor en el navegador.

```
[ SERVER COMPONENT (Page) ]
         │ 
         │ 1. Hace fetch a la Base de Datos o API
         │ 2. Obtiene los datos (e.g., lista de productos)
         ▼
[ CLIENT COMPONENT (ProductList) ] <─── Se los pasa como PROPS
         │
         │ 3. Recibe los productos
         │ 4. Maneja interactividad (filtros, ordenamiento, estado)

```

### El código en la práctica (Enfoque Senior)

Imaginate que tenemos una página de productos. El fetching se hace en el servidor (donde es ultra rápido y seguro), y la interactividad (un buscador dinámico) se maneja en el cliente.

```tsx
// app/productos/page.tsx (Server Component por defecto)
import { ProductFilterList } from '@/components/ProductFilterList';

async function getProducts() {
  const res = await fetch('https://api.mi-ecommerce.com/products');
  if (!res.ok) throw new Error('Error al cargar productos');
  return res.json();
}

export default async function ProductosPage() {
  // 1. Buscamos los datos directamente en el servidor
  const products = await getProducts(); 

  return (
    <main className="p-6">
      <h1 className="text-2xl font-bold">Catálogo de Productos</h1>
      {/* 2. Le pasamos los datos serializados al componente cliente */}
      <ProductFilterList initialProducts={products} />
    </main>
  );
}

```

## 2. El concepto fundamental: La Serialización de Datos

Acá es donde el 90% de los juniors tropiezan y se quedan mirando la pantalla sin entender el error. Cuando le pasás datos de un Server Component a un Client Component, esos datos tienen que cruzar la "red" (el puente entre el servidor Node.js y el navegador). Por lo tanto, los datos **deben ser serializables**.

Next.js convierte esas props a un formato intermedio (similar a JSON pero optimizado para React, llamado *RSC Payload*) para que el navegador pueda reconstruirlos.

### 🚫 Qué NO podés pasar como prop:

* **Funciones:** No podés pasar un `onClick={() => console.log('hola')}` desde un Server Component a un Client Component. Las funciones no se pueden serializar a texto para mandarlas por la red.
* **Instancias de clases complejas:** Si usas un ORM como Prisma o Mongoose, no podés pasar la instancia cruda del modelo si tiene métodos propios de la clase.
* **Objetos `Date` nativos directamente (en ciertas configuraciones antiguas):** Aunque hoy React maneja mejor las fechas, la buena práctica es pasar *timestamps* o strings ISO para evitar problemas de zona horaria entre el servidor y el cliente.

### Qué SÍ podés pasar:

* Tipos primitivos: `string`, `number`, `boolean`, `null`.
* Objetos planos (`{}`) y Arrays (`[]`) que contengan tipos primitivos.

## 3. Patrón Avanzado: Composición (Metiendo Server Components "dentro" de Client Components)

¿Qué pasa si tenés un componente cliente gigante (por ejemplo, un `Sidebar` interactivo que se abre y cierra con un botón) y adentro querés mostrar una lista de usuarios que viene de la base de datos?

Si ponés `"use client"` arriba del `Sidebar` e importás el componente de usuarios adentro, automáticamente transformás todo en cliente, perdiendo los beneficios del servidor. Para resolver esto, los ingenieros de software usamos el **Patrón de Composición utilizando la prop `children**`.

Mirá este truco arquitectónico:

```tsx
// 1. COMPONENTE CLIENTE: Solo le importa la estructura y su estado
// components/Sidebar.tsx
'use client';
import { useState } from 'react';

export function Sidebar({ children }: { children: React.ReactNode }) {
  const [isOpen, setIsOpen] = useState(false);

  return (
    <aside className={isOpen ? 'w-64' : 'w-16'}>
      <button onClick={() => setIsOpen(!isOpen)}>Toggle</button>
      
      {/* Acá se va a renderizar lo que venga del servidor */}
      <div className="content">{children}</div>
    </aside>
  );
}

```

```tsx
// 2. LA PÁGINA (SERVER COMPONENT): Conecta los dos mundos
// app/layout.tsx
import { Sidebar } from '@/components/Sidebar';
import { UserListServer } from '@/components/UserListServer'; // Server Component

export default function DashboardLayout() {
  return (
    <div className="flex">
      {/* Pasamos el Server Component como hijo (children) del Client Component */}
      <Sidebar>
        <UserListServer /> 
      </Sidebar>
      <main>Contenido principal</main>
    </div>
  );
}

```

### ¿Por qué funciona esto y por qué es una maravilla de la ingeniería?

Porque React, en el servidor, renderiza `UserListServer` por completo, genera su HTML/RSC Payload, y luego simplemente se lo "inyecta" al `Sidebar` donde dice `children`. El `Sidebar` puede abrirse, cerrarse y cambiar su estado en el cliente, pero el contenido de adentro se procesó limpiamente en el servidor. El bundle de JS sigue estando optimizado al máximo.

## Enfoque de Mercado: ¿Cómo se maneja esto en proyectos reales?

Hoy en día, las empresas serias evitan el viejo patrón de "traer todo con `useEffect` en el cliente" porque genera interfaces parpadeantes y lentas. La arquitectura moderna dicta:

1. **Server-first:** Traés la mayor cantidad de datos posibles en el Server Component (`page.js`).
2. **Hydrate-second:** Le pasás esos datos como `initialState` a tus componentes interactivos en el cliente si es que el usuario necesita filtrarlos o modificarlos en tiempo real.
